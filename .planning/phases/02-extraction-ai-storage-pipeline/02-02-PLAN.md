  <name>Task 1: Create task extraction prompt and robust JSON parser</name>
  <files>lib/task-extractor.ts</files>
  <action>
Create `lib/task-extractor.ts` with three components:

**1. TASK_EXTRACTION_SYSTEM_PROMPT** (exported const):

Design a system prompt that instructs the LLM to:
- Analyze provided text and extract action items, tasks, meetings, reminders, commitments
- Return ONLY a valid JSON array -- no markdown code fences, no explanation, no preamble
- Each object must have: `type` (one of "call", "meeting", "task", "note", "reminder"), `action` (string), `deadline` (ISO 8601 string or null), `priority` ("low", "medium", "high"), `context` (string with additional details, names, references)
- Return empty array `[]` if no action items found
- Include 2-3 few-shot examples showing correct format (see RESEARCH.md for example format)
- Set temperature to 0.1 in the calling code (not in the prompt) for deterministic output

**2. parseTasksFromLLMResponse(content: string)** function:

Returns an array of partial task objects (without id, sourceUrl, platform, createdAt, txHash, memsyncId, synced -- those are added by the caller).

The parser must handle common LLM output issues:
- **Markdown code fences:** Strip ```json ... ``` wrapping before parsing
- **Extra text before/after JSON:** Try to extract JSON array from within surrounding text using regex match for `\[[\s\S]*\]`
- **Invalid JSON:** Return empty array (never throw)
- **Missing fields:** Use defaults (type: 'task', priority: 'medium', deadline: null, context: '')
- **Extra fields:** Ignore them (only pick known fields)
- **Single object instead of array:** Wrap in array

The return type should be:
```typescript
type RawTask = {
  type: 'call' | 'meeting' | 'task' | 'note' | 'reminder';
  action: string;
  deadline: string | null;
  priority: 'low' | 'medium' | 'high';
  context: string;
};
```

**3. Validation helpers** (exported):
- `validateType(t: string): Task['type']` -- returns the type if valid, 'task' otherwise
- `validatePriority(p: string): Task['priority']` -- returns the priority if valid, 'medium' otherwise

Import the Task type from `@/lib/types` for the type reference, but the RawTask type (return type of parseTasksFromLLMResponse) should be defined locally since it omits most Task fields.

The module must NOT import from opengradient.ts or make any fetch calls -- it is pure data transformation.

**Add a new exported function `extractTasksWithProof`:**

```typescript
export async function extractTasksWithProof(
  x402Fetch: typeof fetch,
  text: string
): Promise<{ rawTasks: RawTask[]; txHash: string | null }>
```

This function:
1. Calls the OpenGradient LLM endpoint (same `OG_LLM_ENDPOINT` constant) via the x402-wrapped fetch.
2. Uses `anthropic/claude-4.0-sonnet` as the model (per AI-02 requirement). Add a comment noting GPT-4o as proven fallback if Claude is unavailable.
3. Sets `X-SETTLEMENT-TYPE: individual` (NOT `settle-batch` used in testLLMCall). This records full input/output metadata on-chain for TEE attestation (satisfies AI-05, AI-06, AI-07).
4. Uses the `TASK_EXTRACTION_SYSTEM_PROMPT` from `@/lib/task-extractor`.
5. Sets `max_tokens: 2000` and `temperature: 0.1`.
6. After receiving the response, extracts the transaction hash from the `X-PAYMENT-RESPONSE` header:
   - Get header value: `response.headers.get('x-payment-response')`
   - Base64-decode it: `JSON.parse(atob(paymentResponse))`
   - Extract `decoded.transaction` as the txHash
   - Wrap in try/catch -- if decode fails, txHash is null (non-fatal)
7. Parses the LLM response body as OpenAI chat completions format: `data.choices?.[0]?.message?.content`
8. Passes the content through `parseTasksFromLLMResponse` from task-extractor.ts.
9. Returns `{ rawTasks, txHash }`.

Import `TASK_EXTRACTION_SYSTEM_PROMPT` and `parseTasksFromLLMResponse` from `@/lib/task-extractor`. Import `RawTask` type from task-extractor.ts as well (export it from there).

The function should NOT handle storage, task ID generation, or enrichment -- it only does LLM inference and returns raw results. The background.ts handler (Plan 02-03) will enrich and store.

Keep the existing `testLLMCall` function unchanged -- it uses `settle-batch` and `gpt-4o` intentionally (cheap test call).
